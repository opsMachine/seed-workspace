# transcript-vault-mcp — Architecture

How the pieces fit together, where data lives, and why the system is structured the way it is.

---

## Overview

The system is a three-phase offline pipeline feeding a read-only MCP server. Each phase produces artifacts consumed by the next; nothing in the serving layer writes back to the pipeline.

```
Recorder API
     │
     ▼
[extract]  ──→  data/raw/*.json           (one JSON file per meeting)
     │
     ▼
[transform] ──→  data/index.db            (SQLite + FTS5, label projections)
            ──→  vault/Meetings/*.md      (human-browseable transcripts, gitignored)
            ──→  vault/People/*.md        (opaque meeting refs, gitignored)
            ──→  vault/Companies/*.md     (opaque meeting refs, gitignored)
     │
     ▼
[embed]     ──→  data/vectors/            (LanceDB, 384-dim local embeddings)
     │
     ▼
[mcp-server]     reads index.db + vectors/, serves 12 tools + 4 prompts over stdio
```

---

## Phase 1 — Extract

**Entry point:** `src/extract.ts`  
**State:** `data/sync-state.json` (cursor + deduplicated `downloaded_ids` set)

The extractor is recorder-agnostic. It loads the adapter named by `RECORDER_ADAPTER` (default: `fathom`), then runs the same dedup-and-checkpoint loop regardless of recorder. Each meeting is saved as `data/raw/<recording_id>.json`.

Resumable: on interrupt or failure, re-running picks up from the last checkpoint. State is shared across adapters — switching recorders requires deleting `sync-state.json` first to avoid recording_id namespace collisions.

**Adapter contract** (`adapters/index.ts`):

```typescript
interface RecorderAdapter {
  fetchMeetings(state: SyncState): AsyncGenerator<Meeting>;
}
```

A single `AsyncGenerator` method. The adapter handles pagination, rate limiting, and auth; the extractor handles state, dedup, and disk writes. Fathom is fully implemented (throttled at 55 req/min with 429 backoff). Fireflies, Otter, and Granola ship as stubs.

## Phase 2 — Transform

**Entry point:** `src/transform.ts`  
**Inputs:** `data/raw/*.json`, `config/labels.yml`  
**Outputs:** `data/index.db`, `vault/Meetings/`, `vault/People/`, `vault/Companies/`

Transform has two jobs:

**1. Build the SQLite index (`data/index.db`).** The schema has tables for meetings, participants, transcript chunks, and label projections. FTS5 virtual tables cover meeting titles, summaries, and chunk text for keyword search. Label data from `config/labels.yml` is projected into the participant and meeting tables at transform time so the MCP server never has to parse YAML at query time.

**2. Write the vault markdown.** Each meeting gets a `vault/Meetings/<recording_id>.md` with full transcript text. People and Company files render meeting histories as opaque numbered links (`[Meeting #12345](../Meetings/12345.md)`) — titles and dates deliberately don't appear in these filenames or links so the human-browseable vault doesn't leak relationship context through git history.

Transform is idempotent and re-runnable. `apply-labels` calls it after merging new label decisions.

---

## Phase 3 — Embed

**Entry point:** `src/embed.ts`  
**Model:** `Xenova/all-MiniLM-L6-v2` via `@xenova/transformers` (runs locally, no API key)  
**Output:** `data/vectors/` (LanceDB table, 384-dim float32 vectors)

Each meeting is chunked into overlapping segments (`src/lib/chunk.ts`) covering transcript turns and summary sections. Chunks are embedded in batches of 32 and flushed to LanceDB every 256. The process is resumable: existing chunk IDs are loaded on startup and skipped.

`--fresh` (or `FRESH=1`) drops the table and re-embeds everything — useful after a model change or major data restructure.

**Why a local model instead of an API?** Transcripts contain sensitive relationship and business information. Sending them to an embedding API for indexing is a meaningful data exposure. The local `all-MiniLM-L6-v2` model (22M params, ~90MB) gives good-enough retrieval quality for this use case at zero marginal cost per query.

## The labeling loop

Labels are a human-in-the-loop layer that sits between transform and the MCP server. The YAML file is the single source of truth; the generated vault files are derived from it.

```
config/labels.yml          ← TRACKED — the only labeling artifact in git
        ▲
        │   npm run apply-labels (atomic merge → re-runs transform)
        │
data/proposals.yml         ← gitignored, written by AI chat
        ▲
        │   AI chat reads dossier, writes decisions
        │
data/candidates/*.md       ← gitignored, generated by prep:people / prep:companies / prep:calls
```

**Flow:**

1. `npm run prep:people` triages participants into confidence buckets (A_PROPOSE, B_REVIEW, C_SKIP) based on meeting frequency, domain, and existing label coverage. Outputs a dossier to `data/candidates/people-dossier.md`. Idempotent: skips anyone already in `labels.yml` or `proposals.yml`.
2. AI reads the dossier in chat and writes tag decisions into `data/proposals.yml`.
3. `npm run apply-labels -- --dry-run` previews the diff.
4. `npm run apply-labels` merges `proposals.yml` into `labels.yml` atomically, then re-runs transform so the SQLite projections stay in sync.

Run companies before people — the person dossier uses company tags as a prior for relationship inference. Auto-derived call types (`internal`, `advisory`) are computed at transform time and never appear in the YAML.

## The MCP server

**Entry point:** `src/mcp-server.ts`  
**Transport:** stdio (MCP standard)  
**Server name:** `transcript-vault-kb`

The server is stateless across requests — it opens fresh database connections per tool call and delegates all computation to `src/lib/search.ts`. No in-memory state other than the connection pool.

**Search layer** (`src/lib/search.ts`): all 12 tool implementations live here. The two search modalities:

- **Keyword (`keyword_search`):** FTS5 `MATCH` queries against the SQLite virtual table, supporting `AND`, `OR`, `NOT`, quoted phrases. Fast, exact.
- **Semantic (`semantic_search`):** embeds the query with the same local model used at index time, runs an ANN query against LanceDB, joins results back to SQLite for metadata. Supports a `min_distinct_meetings` parameter to prevent a single verbose meeting from dominating results.

**`verify_claim` — dual retrieval:** The claim verifier runs two retrievals — one for the claim as stated, one for the counter-claim (either supplied or heuristically negated). Results are bucketed into `SUPPORTS`, `CONTRADICTS`, and `UNCLEAR` based on semantic distance thresholds and negation-proximity scoring. This is the tool that enables a research session to actively look for disconfirmation rather than just confirmation.

**`get_entity_timeline` and `get_project_status`:** Timeline retrieval orders all mentions of a named entity by `meeting_date` rather than retrieval score — solving the common failure mode of status queries that surface an old high-relevance mention instead of the most recent one. `get_project_status` is a thin wrapper that picks the most recent timeline entry and formats it as a status.

**4 prompts:** `client_persona`, `sales_coaching`, `content_from_meetings`, `account_prep`. These are MCP prompt objects — they return structured message arrays that prime the AI with a research workflow before it starts making tool calls, rather than expecting the AI to invent the right sequence.

**Resource:** `skill://transcript-research` — serves the `skills/transcript-research/SKILL.md` research methodology as a readable MCP resource. Any connected client can fetch it directly; Cursor and Claude Code also discover it as an IDE skill.

## Filesystem layout under `VAULT_DATA_ROOT`

```
<VAULT_DATA_ROOT>/          (defaults to repo root if unset)
  data/
    raw/                    ← gitignored: JSON-per-meeting from extract
    index.db                ← gitignored: SQLite + FTS5
    vectors/                ← gitignored: LanceDB
    sync-state.json         ← gitignored: extract cursor + dedupe set
    proposals.yml           ← gitignored: pending label decisions
    candidates/             ← gitignored: generated dossier markdown
    logs/                   ← gitignored: nightly-sync output
    .sync.lock              ← gitignored: flock guard
  vault/
    Meetings/               ← gitignored: full transcript markdown
    People/                 ← gitignored: generated from labels.yml
    Companies/              ← gitignored: generated from labels.yml
  config/
    labels.yml              ← TRACKED: taxonomy source of truth
```

`VAULT_DATA_ROOT` can point anywhere — inside or outside the seed-workspace. **Note:** the seed-workspace `.gitignore` only protects paths inside the workspace. If `VAULT_DATA_ROOT` is set to an external path, ensure that location is itself untracked or in a separate gitignored directory.

## Design decisions worth noting

**Why SQLite + LanceDB rather than a single vector store?**  
Keyword and semantic search serve different queries. Precise terms (names, acronyms, project codes), FTS5 is faster and exact. Conceptual questions ("what has this person said about pricing pressure?"), semantic search is better. The two modalities compose — most complex research queries use both. Maintaining two stores is the cost of having both available cleanly.

**Why local embeddings?**  
Transcripts contain relationship data, business strategy, and personal context. Routing them through an external embedding API creates a data exposure that compounds with every new meeting. `all-MiniLM-L6-v2` is good enough for the retrieval quality needed here, runs in milliseconds on CPU, and costs nothing per query.

**Why opaque links in vault/People/ and vault/Companies/?**  
People and company files use `[Meeting #12345](../Meetings/12345.md)` links instead of meeting titles and dates. Meeting titles often contain client names, project names, and relationship context — that information leaking through git history (even in a private repo) is avoidable with a one-time design choice.

**Why is the labeling loop human-in-the-loop rather than automated?**  
Relationship tags (`client`, `advisor`, `prospective_client`, etc.) determine how the AI frames every query about a person. An automated classifier making these wrong in bulk would silently corrupt the research layer. The human review step is the correctness gate for the most consequential metadata in the system.
